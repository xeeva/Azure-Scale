The Human Cost

Microsoft 365 exposes 180,263 configurable properties through the Graph API alone. At a conservative estimate of 5 minutes per property to read, understand, and evaluate each setting, the total time to comprehend the full M365 configuration surface is:

  • 15,022 hours of focused study
  • 1,878 working days (at 8 hours/day)
  • 7.2 years of full-time work (at 260 days/year)
  • Equivalent to roughly 1,351 pages of dense technical documentation

And this only counts what Graph exposes. Exchange Online, Teams, and SharePoint each have additional admin-level settings only accessible through PowerShell modules, adding hundreds more configuration surfaces not counted here.

The most complex workloads by property count are Entra ID (71,776), Security (21,776), OneDrive (20,505). But complexity is not just depth — it is also the interconnection between workloads. A single Conditional Access policy in Entra ID may reference users, groups, applications, device compliance from Intune, named locations, and authentication methods — each with its own property tree and licensing requirements.

34 of 35 workloads have distinct licensing tiers, each requiring separate evaluation for feature availability, cost, and compliance.

In many organisations, a single M365 administrator is expected to understand all of this — not superficially, but well enough to configure Conditional Access policies, manage Intune device compliance, set up DLP rules in Purview, troubleshoot mail flow in Exchange, and answer security audit questions. The 180,263 properties and 16,147 API operations are not static. They grow with every Graph API update, every new M365 feature, every license tier change.

Methodology

Data Source

The Microsoft Graph OpenAPI specification (v1.0), a single monolithic YAML published by Microsoft at microsoftgraph/msgraph-metadata on GitHub. No authentication or tenant access required.

Workload Discovery

API paths are grouped into workloads by OpenAPI tags and path prefixes. Tags like users, groups, applications map to Entra ID; deviceManagement to Intune; etc.

Operations Count

Every HTTP operation (GET, POST, PATCH, DELETE) on every path is counted. Read vs. write operations are tracked separately.

Config Properties

Parsed from components/schemas. Each entity type's schema is recursively walked to count every leaf property. Circular references are detected and broken. allOf, oneOf, anyOf compositions are resolved.

Licensing Options

Curated mapping of 34 workloads to applicable M365 license tiers: Free, E1/E3/E5, P1/P2, Business plans, and standalone add-ons.

Complexity Score

Weighted composite: Config properties (×2.0), Entity types (×1.0), Operations (×0.5), Licensing (×3.0).

Scope note: This report measures everything exposed through the Microsoft Graph API (v1.0). Graph has strong coverage of Entra ID, Intune, Planner, and OneDrive. Coverage of Exchange Online, Teams, and SharePoint admin settings is partial — many admin-level configurations are only accessible via PowerShell modules (Exchange Online Management, Teams PowerShell, SharePoint Online Management), not Graph. Those will be added in a future Tier 2 analysis.

Below is the complete, workload-by-workload breakdown of every service area exposed through the Microsoft Graph API. Each collapsible section represents one workload — a logical grouping of related entity types and API operations.

  • Entity Types — the Graph schema objects (users, devices, policies, etc.) that you create, configure, and manage. Each has its own property set, relationships, and API surface.
  • Configuration Properties — the total number of individual settings found by recursively walking the OpenAPI schema for each entity type. This is the true "knob count".
